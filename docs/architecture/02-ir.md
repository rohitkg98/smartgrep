# IR — Intermediate Representation

**File:** `src/ir/types.rs`

The IR is the contract between parsers and the index builder. A parser reads source code and produces an `Ir`. The index builder reads that `Ir` and knows nothing about how it was produced — whether from Rust, Go, or Java.

## The top-level type

```rust
pub struct Ir {
    pub symbols: Vec<Symbol>,
    pub dependencies: Vec<Dependency>,
}
```

That's it. Two flat lists. No trees, no nesting, no language-specific fields.

## Symbol

A `Symbol` is any named structural unit in source code.

```rust
pub struct Symbol {
    pub name: String,              // short name: "parse_file"
    pub qualified_name: String,    // fully qualified: "parser::rust::parse_file"
    pub kind: SymbolKind,          // Function, Struct, Trait, Impl, ...
    pub loc: SourceLoc,            // file, line, col
    pub visibility: Visibility,    // Public, Crate, Private
    pub signature: Option<String>, // full signature string
    pub parent: Option<String>,    // enclosing type (for methods)
    pub attributes: Vec<String>,   // #[derive(...)], #[test], etc.
    pub fields: Vec<Field>,        // struct/enum fields
    pub params: Vec<Param>,        // function parameters
    pub return_type: Option<String>,
}
```

`SymbolKind` covers: `Function`, `Method`, `Struct`, `Enum`, `Trait`, `Impl`, `Const`, `TypeAlias`, `Module`.

`SourceLoc` is just a file path plus line and column number.

## Dependency

A `Dependency` is a directed edge from one symbol to another.

```rust
pub struct Dependency {
    pub from_qualified: String, // qualified name of the source symbol
    pub to_name: String,        // name of the target (may not be qualified)
    pub kind: DepKind,          // how the dependency arises
    pub loc: SourceLoc,         // where in source it appears
}
```

`DepKind` covers: `Import`, `FunctionCall`, `TypeReference`, `TraitImpl`, `FieldType`.

## Why a flat list instead of a tree?

Source code is hierarchical (modules contain functions, impls contain methods). The IR flattens everything. This makes the index builder simple — it just iterates over slices — and makes queries uniform. Hierarchy is preserved through the `parent` field on `Symbol` and `from_qualified` / `to_name` on `Dependency`.

## The contract in practice

```
Parser produces:          Index builder sees:
  Ir {                      for sym in ir.symbols { ... }
    symbols: [...],         for dep in ir.dependencies { ... }
    dependencies: [...],
  }
```

The builder does not import any parser module. The parsers do not import any index module. They share only `src/ir/types.rs`.
